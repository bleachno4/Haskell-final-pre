# 四人代码分工与工作量说明

核心原则：四个成员都要负责代码，不能让某一个人只负责选题、背景或总结。背景和选题只作为开场的一小部分，主要展示重点放在 Haskell 知识如何落到 EDA demo 的代码里。

## 总体拆分方式

这个项目可以按一条 EDA 前端流水线拆分：

```text
文本输入
  -> Parser
  -> AST
  -> 求值和真值表
  -> 化简和等价性检查
  -> Netlist
  -> Verilog-style 输出
```

为了让四个人工作量均衡，建议把 parser、AST、仿真验证、转换输出拆成四个代码模块。每个人既讲 Haskell 知识点，也讲自己负责的代码片段。

## 成员 A：输入语言与 Parser

对应文件：`src/CircuitEDA/Parser.hs`

代码责任：

- `Parser` 类型：`newtype Parser a = Parser { runParser :: String -> Maybe (a, String) }`
- `Functor`、`Applicative`、`Monad`、`Alternative` 实例
- `parseDesign`
- `parseExprText`
- `exprParser`
- `parseOr`、`parseXor`、`parseAnd`、`parseNot`、`parseAtom`
- `keyword`、`identifier`、`symbol`
- `examples/bad_syntax.logic`

展示责任：

- 用 1 分钟说明题目定位：为什么 EDA 前端需要“文本 -> 结构”。
- 重点讲 parser combinator：小 parser 如何组合成完整表达式 parser。
- 解释优先级：`NOT` 高于 `AND/NAND`，再高于 `XOR`，最后是 `OR/NOR`。
- 展示错误输入：`out = (a AND) OR b` 如何触发 parse error。

Haskell 知识点：

- `newtype`
- `Maybe`
- `Functor`
- `Applicative`
- `Monad`
- `Alternative`
- parser combinators

工作量评估：

- 代码阅读和讲解难度：中等偏上。
- 需要讲清楚抽象类型和组合思想。
- 预计工作量：约 25%。

## 成员 B：AST、ADT 与结构分析

对应文件：`src/CircuitEDA/AST.hs`

代码责任：

- `Expr`
- `Assignment`
- `Design`
- `pretty`
- `vars`
- `designInputs`
- `designOutputs`
- `gateCount`
- `depth`
- `renderTree`

展示责任：

- 说明为什么布尔电路适合用 AST 表示。
- 讲 ADT 如何把 `Lit`、`Var`、`Not`、`And`、`Or`、`Xor`、`Nand`、`Nor` 表示成不同构造器。
- 展示 `renderTree` 的 AST 输出。
- 解释递归如何统计变量、门数量和深度。

Haskell 知识点：

- algebraic data types
- record syntax
- pattern matching
- recursion
- deriving
- list processing

工作量评估：

- 代码相对直观，但需要把“电路结构”和“Haskell 类型”对应起来。
- 预计工作量：约 24%。

Haskell 知识点：

工作量评估：

- 预计工作量：约 27%。

## 成员 D：化简、Netlist 与 Verilog-style 输出

对应文件：`src/CircuitEDA/Transform.hs`

代码责任：

- `simplify`
- `simplifyOnce`
- `simplifyDesign`
- `SimpleState`
- `fresh`
- `emit`
- `renderNetlist`
- `renderVerilog`

展示责任：

- 讲模式匹配如何实现逻辑化简规则。

- 解释为什么生成 netlist 需要状态编号。
- 讲 `SimpleState s a` 的核心类型：`s -> (a, s)`。
- 展示 Verilog-style `assign` 输出。
- 负责现场集成运行和备用输出文件。

Haskell 知识点：

- pattern matching
- recursion
- 自定义 State monad 思想
- 纯函数代码生成
- IO 和脚本集成

工作量评估：

- 内容跨度较大，但 parser 已经从 D 中拆给 A，所以工作量更平衡。
- 预计工作量：约 24%。

## 一周时间安排建议

第 1 天：四个人一起跑通 demo，确认每个人能编译和运行 `.\run_demo.ps1`。

第 2 天：A 重点读 parser；B 重点读 AST；C 重点读 eval/truth table；D 重点读 simplify/netlist/verilog。

第 3 天：每个人整理自己负责的 2 到 3 个关键代码片段。

第 4 天：每个人补充一个对应 demo：A 展示 bad syntax，B 展示 AST tree，C 展示 full adder truth table，D 展示 simplify 和 Verilog-style 输出。

第 5 天：合并讲稿，统一术语，不再强调“某个 Wikibook 专题”，而是强调 Haskell 知识点如何解决 EDA 前端问题。

第 6 天：完整彩排，检查每个人是否都能回答自己代码模块的问题。

第 7 天：压缩时间，准备备用方案：如果现场不能编译，就打开 `outputs/demo_output.txt`。

## 工作量是否均衡

整体上是均衡的：

| 成员 | 代码复杂度 | 展示压力 | 预计占比 |
|---|---|---|---|
| A | 中等偏上，parser 抽象性强 | 开场 + parser，压力中等 | 25% |
| B | 中等，结构最清楚 | 需要把 AST 讲形象 | 24% |
| C | 中等偏上，涉及正确性 | 老师可能追问最多 | 27% |
| D | 中等偏上，涉及输出和集成 | 负责现场运行，压力较高 | 24% |

这样分工的好处是：每个人都能说“我负责了项目中的一个代码模块”，而不是有人只负责背景介绍。展示时也更像一个完整工程：A 负责输入，B 负责表示，C 负责语义，D 负责转换和输出。
